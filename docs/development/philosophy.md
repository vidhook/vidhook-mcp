# 開発フィロソフィー

このリポジトリで開発判断を行う際の基本原則。実装・レビュー・議論のすべての場面で参照する判断基準とする。
プロジェクト固有の判断基準が別途ある場合はそちらを優先する。

## DRY (Don't Repeat Yourself)

同じ知識・ロジックを複数箇所に書かない。重複は **意図せず分岐する温床** であり、修正漏れと品質劣化の主要因となる。

- 共通ロジックは関数・モジュールに抽出する
- 同じ概念を表す型・定数は1箇所に定義し、参照で使う
- ただし「形が似ているだけ」のコードを早すぎる抽象化で統合しない（**Rule of Three** に近い感覚で判断）

## YAGNI (You Aren't Gonna Need It)

「将来必要になるかもしれない」を理由にコードを書かない。**必要が顕在化してから書く**。

- 使われていない引数・フラグ・抽象化レイヤーを「拡張性のため」に残さない
- インターフェース抽出は実装が複数現れた時点で行う（Rule of Three）
- 想像上の要件は仕様の負債になる

## 後方互換性は目的化しない

不要な分岐・互換シムを残さない。**余計な分岐はメンテナンス性を低下させる**。

- 古い API・型・関数を使う箇所がなくなったら、即座に削除する
- マイグレーション期間用のコードは目的を達したら除去する
- 「念のため残しておく」は禁止。使われていないなら消す
- リネーム時に旧名のエイリアスを残さない（IDE のリファクタリング機能で一括変更する）

ただし、外部公開 API（他システムが依存しているもの）は別観点で扱うこと。

## テストのモックは最小限にする

モックの濫用は「実装の形」をテストすることにつながり、リファクタ耐性を下げる。
**モックは外部システムとの境界に限定する**。

- **モックして良いもの**: 外部システムとの境界（外部 API、外部ストレージ、外部認証基盤、メールサービス、決済等、自プロジェクトの外側との通信）
- **モックしないもの**: 自プロジェクト内の関数・クラス・モジュール
- DB アクセスを伴うテストは可能な限りテスト用の実 DB 環境を使う
- 外部境界のモックは1か所に集約し、各テストで個別にスタブを書かない

## 外部境界の正しさはモックで担保しない

モックは「自分が書いた仕様通りに動く」を検証するもので、「**外部システムが本当にその仕様か**」は検証できない。
**外部境界モジュールには、実環境で1度疎通させた証拠**（リクエスト/レスポンスのログ等）を必ず残す。

- 新規追加した外部依存（ライブラリ・API・MCP サーバ・SDK 等）の使用方法は、実装前に公式ドキュメントを参照する（Context7 等）
- 実装後、実環境で1度叩いた証拠を残す（不可能な場合は理由を記録）
- 完全モックの単体テストが通っただけで「完了」と判定しない

## 常にコードを見渡し、リファクタリングを行う

実装は「動けば終わり」ではない。**周辺コードを見渡し、改善の余地があればその場で直す**。

- 機能追加時、関連する既存コードに重複・冗長・古い設計が残っていないか確認する
- 機会があるたびに小さくリファクタする（「今度直そう」は溜まる）
- ただしスコープの肥大化に注意。本来のタスクの完遂を優先し、大きな変更は別 Issue として切り出す

## 関心の範囲外の課題を見つけた場合は Issue 化する

リファクタ中・実装中に「これは別件だが直すべき」と気づいたら、その場では深追いせず Issue として記録する。

- 即座に直すのは **本来のタスクと密結合な場合のみ**
- 独立した課題はプロジェクトで採用している Issue 起票手順で記録する
- 「忘れずに後で」は最も失われやすい情報。発見したらすぐ記録する

## 根本原因に対処する

エラー・テスト失敗・予期しない挙動に出会ったとき、**症状を消すのではなく原因を特定する**。

- テストが失敗したら `--no-verify` で回避しない
- 例外を握りつぶす try/catch を「とりあえず」で書かない
- 警告を抑止オプションで黙らせない
- 環境固有の問題に見えても、まず再現条件を切り分ける
